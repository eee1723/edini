🎯 设计原则 1：分层职责

VEX/Python 只生成骨架（点序列、闭合轮廓、点属性）。
原生节点 保证封闭性（Sweep endcaptype、PolyExtrude output_back、Cap）。
LLM 调用预制函数而非手写脚手架代码。
CTP 实例化组件，可独立替换。

🎯 设计原则 2：封闭性由构造保证

手写 createPolygon 需要每个面的 winding 都正确——LLM 不可靠。
一个闭合曲线 + polyextrude(output_back=1) → 自动生成正确的封闭体。
一个路径 + 截面 + sweep(endcaptype=1) → 自动封端。
让 SOP 做它擅长的事；让 LLM 做它擅长的事。

🎯 设计原则 3：确定性优先

每个 vexlib 函数都是测试过的、有明确输入输出的"积木"。
LLM 调积木，不现场烧砖。
向后兼容：旧 recipe 继续工作，新字段可选。
Builder 确定性装配网络——agent 只写组件代码，从不写 createNode/wiring。

路由 决策路由器

SKILL.md 的核心职责：在 LLM 开始写任何代码之前，将任务路由到正确的后端和构建路径。

Step 1 — 选择后端

SKILL.md § Step 1
默认首选 VEX 骨架 + Sweep/PolyExtrude，而不是 Python SOP 手写几何。
这是从楼梯实验数据中得出的关键结论：
Python SOP 手写 createPolygon → 系统性封闭失败（nm=48+）。
VEX 骨架 + Sweep → 封闭性由 SOP 保证，零失败。
# 🧭 后端选择决策流 ✅ 【默认首选】VEX 骨架 + Sweep/PolyExtrude（vexlib） 适用：管、梁、柱、楼梯、墙、栏杆、框架 → 任何"路径+截面"或"轮廓+拉伸"可表达的形体 ⚠️ 【备选】Python SOP 手写几何 仅当形体无法用路径+截面或轮廓+拉伸表达时 如：复杂分支拓扑、递归结构、文件IO、HDA逻辑 如果发现自己在写 createPolygon + winding 表 → 立即重构 🔧 【后处理】VEX wrangle（逐元素） 噪声位移、方向随机化、属性驱动散布、变形 📦 【辅助】原生节点链 仅后处理：PolyBevel、Subdivide、Normal、Fuse、Clean

铁律 四大铁律

这些规则在 SKILL.md 中说一次，全文档引用。LLM 不得重新推导。

🔴 铁律 1：Python SOP cook body 内禁止 createNode

cook body 调用 hou.node(...).createNode(...) 会重新进入 cook 求值 → 无限递归。cook body 只能在自己的 geo 上 createPoint/createPolygon/addAttrib。要构建网络，使用 network_mode=true（代码在容器中运行，不是 cook 内）。

🔴 铁律 2：用 houdini_node_parms(type) 查询参数名

参数名在 Houdini 版本间漂移。Normal SOP 的尖角参数是 cuspangle 不是 cangle/cusp。构建 harness 硬校验 postprocess 参数名——猜错的名字是构建期错误，不是静默丢失。

🔴 铁律 3：每个几何部件必须标记 @component_id

创建几何之前声明：geo.addAttrib(hou.attribType.Prim, "component_id", "")。每个部件的每个多边形上：poly.setAttribValue("component_id", "<id>")。没有它，verify_orientation 无法运行，commit_sandbox 拒绝提交。

🔴 铁律 4：封闭性由构造保证，不靠手写 winding ★新

手写的 createPolygon 需要推理每个面的顶点顺序——LLM 不擅长。一个闭合轮廓 + polyextrude(output_back=1) 或 路径+截面 + sweep(endcaptype=1) 由 SOP 自动保证封闭。始终优先选择 SOP 保证的路径。如果必须手写多边形，用 add_box 辅助函数（已提供经过验证的 winding 表）。

Recipe Schema（组件声明）

# 📦 组件代码规则 每个组件的 code 是单独 Python SOP 的 cook body： 1. 读自己的几何：node = hou.pwd(); geo = node.geometry() 2. 创建几何之前声明属性： geo.addAttrib(hou.attribType.Prim, "component_id", "") 3. 每个面标记： poly.setAttribValue("component_id", "") 4. 禁止调用 createNode → builder 拥有网络所有权 Builder 自动做的事（agent 不写）: ┌─────────────┐ │ _python │ ← 你的代码 └──────┬──────┘ ┌──────┴──────┐ │ _anchors│ ← builder 生成锚点 └──────┬──────┘ ↓ copy_ (copytopoints) ← 自动接线 ↓ _idfix (覆盖 component_id) ↓ merge_all → postprocess... → OUT builder 后检查 component_id 存在性， 报告缺失的 id（component_id_check.missing）。

Anchor + 参数关联

params-and-linkage.md
锚点支持 position_expr 表达式："wheelbase/2" 引用资产级参数。
表达式引擎是安全沙盒——只允许算术和数学函数白名单。
构建时求值，组件形状保持动态（ch() 驱动）。
# 🔗 参数关联提示词 资产级参数声明（在 recipe 顶层）： "params": { "wheelbase": {"default": 1.0, "min": 0.5, "max": 2.0}, "wheel_r": {"default": 0.35} } 组件代码中读取（相对通道引用）： wheelbase = hou.ch('../wheelbase') # 往上一级 = sandbox root wheel_r = hou.ch('../wheel_r') 锚点位置用表达式（构建时求值）： "position_expr": ["wheelbase/2", "wheel_r", "-0.55"] 支持的表达式语法： 参数名、算术（+ - * / % **）、一元 -、 数学函数白名单（sin cos sqrt abs min max ...）、 常量 pi/e/tau。 其他全部拒绝——这是安全沙盒，不是 Python eval。 修改参数 → 自动重新 cook 所有依赖组件（通道依赖）。

Variant Scatter（变体散布）

declarative-builder.md § Variant Scatter
Houdini 21 验证过的架构：
无 Pack 节点（Copy 在解包源上按 prim variant 属性分发）
attribfrompieces 分配 variant 到散布点
resettargetattribs 按钮转移 per-instance id
# 🎲 变体散布提示词 何时使用（不是单模板 CTP）： 重复部件有 2+ 种可互换样式（3种窗户、4种树、2种门） → 打破"程序化重复"感 → 提升细节到 Level 3+ 工作流（H21 验证）： variants（prim variant int 标签）→ merge（不解包） scatter_points（只输出 @P，给 i@id） attribfrompieces（pieceattrib=variant, seed） copytopoints 2.0（useidattrib=1, idattrib=variant） resettargetattribs 按钮转移 i@id idfix → OUT component_id = {variant_id}_{id} 为什么这样架构（H21 实测结论）： ✗ Pack By Name 会将 variant 隐藏在 PackedFragment 内 → Copy 的 piece 分发无法读取 → 分发塌陷 ✓ 不解包源 → Copy 正确按 prim variant 分发 ✓ attribfrompieces 可靠覆盖所有 variant（低点数也覆盖） ✓ resettargetattribs 按钮自动填充转移（无需手设多重参数） seed 控制可复现性 —— 同一 seed + 同一散布拓扑 → 跨运行产生相同 variant 分布。

验证 两层验证协议

两层权威验证，不依赖视觉判断。截图仅存档，不做验证回路。

Layer 1：几何健康

houdini_inspect_geometry_health
阻塞（门控 overall_ok + commit）：orphan_points, open_curves
建议（报告但不阻塞）：degenerate_prims, nonmanifold_edges, open_boundary_edges, coincident_points
仅修复阻塞检查；建议检查按需处理。
# 🏥 几何健康检查提示词 MANDATORY — 方向/视觉验证之前必须运行。 跳过它会让非流形边/退化面流入 Boolean/Sweep 并静默破坏结果。 overall_ok 反映仅阻塞检查（orphan_points, open_curves）。 overall_ok=False → 必须修复指定的阻塞检查。 建议检查（non-manifold/边界边/退化）不阻塞—— 但 non-manifold > prim_count 的 20% 暗示结构性重叠， 需要在源头修复（inset/offset 间隙或 Boolean union）。 open_boundary_edges > 0 是开放曲面的预期结果 （地形、单面板、有意开口）——不要试图清零。

Layer 2a：方向验证（权威门）

houdini_verify_orientation
每个方向断言：component_id + kind (radial/elongated/planar) + expected_axis
首选用 construction_axis（确定性，绕开 PCA 估计）。
失败 → 在组件源代码中应用 hint 四元数，不在几何上后旋转。
# 🧭 方向验证提示词 每个组件一个断言： - component_id: 匹配 @component_id 值 - kind: radial（轮/齿轮）| elongated（管/梁/塔）| planar（板/面板） - expected_axis: X/Y/Z/-X/-Y/-Z 首选（B-station）：加 construction_axis construction_axis 是组件生成围绕的局部空间轴。 构建器从锚点 @orient 四元数确定性推导世界轴—— 纯代数，不采样，不 PCA。 构建器也会捕获自洽错误：如果你声明的 construction_axis + 锚点 orient + expected_axis 互相矛盾，构建在 cook 之前拒绝。 ⚠️ Bug 6 已知：细长圆柱体（塔）用 kind=radial 会被误判 90°。塔和柱用 kind=elongated， 或者设置 construction_axis 绕开 PCA。 失败时的修复： hint 字段包含精确的四元数—— 在生成器代码内部应用，不在几何上后旋转。

Layer 2b：组件清单（权威存在性）

houdini_geometry_inventory
列出每个 @component_id：prim/point 计数、包围盒、相对大小。
prim_count > 0 → 组件存在，无论截图看起来怎样。
size_fraction < 0.08 → 太小需要近景捕获。
# 📊 清单是权威的 判断"组件 X 是否存在"时： ✓ prim_count > 0 → 存在。不要重建。 ✗ 截图看起来空 → 忽略。清单是权威的，像素不是。 标记为 SMALL 的组件（size_fraction < 0.08）： 存在但太小，在全资产视图中无法判断。 用 houdini_capture_component_detail 做近景捕获。 修理前后对比清单： 如果组件的 prim_count/包围盒没变， 你的修改没生效——不要再次重建并期望不同结果。 Debug Discipline 反无效循环规则： 1. 修复前先命名缺陷（指定 component_id 或健康检查字段） 2. 每轮只修一个东西 3. 对比清单前后差异 4. 同样缺陷存活 2 次针对性修复 → 方法错了 → 换后端或问用户 5. 重建是最后手段（仅当阻塞性健康检查显示根本性损坏）
