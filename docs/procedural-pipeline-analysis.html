1系统概览

程序化生成系统的核心设计哲学：把易错的命令式操作从 agent 手里拿走，换成确定性建网 + 分层门禁验证。

设计原则： 确定性层（几何健康 / 朝向 / 库存）权威且廉价，视觉层仅作 advisory—— 这避免了"用昂贵的视觉模型做它不擅长的朝向判断"。agent 永远先走 cheap authoritative 层， 只在无法决断时才升级到视觉。

提示词：SKILL.md Query Parameter Names · Forbidden #4 · Harness Rules · Workflow step 2

建网生成（三选一） houdini_build_procedural_asset
① 首选：声明式 Builder —— agent 只写每组件纯几何代码 + recipe，harness 确定性建网（组件 SOP → 锚点 → Copy-to-Points → merge → postprocess → OUT）。
② 回退：network_mode —— agent 手写 createNode 网络（非标准拓扑）。
③ 单件：单 SOP cook —— 纯单件生成器。
提示词：SKILL.md Code-First · Forbidden #1(递归) · #2(绕沙盒) · Network Mode · harness.ts promptGuidelines
⚠ 建网后立即检查：structure_advisory.is_monolithic
▼
4
三层验证（核心防线） houdini_inspect_geometry_health
Layer 1 几何健康（权威·强制）：孤儿点、退化面、非流形边、开放边界——不通过禁止继续。
Layer 2a 朝向（权威·门禁）：PCA / construction_axis 按组件检测轴向。
Layer 2b 库存（权威·数据）：component_id 清单 + prim 数 + 相对大小。
Layer 3 视觉（advisory）：4 视图截图 → describe_image 交叉验证。
提示词：SKILL.md Three-Layer Verification Protocol · Debug Discipline
🔒 commit 时双硬门：structure + orientation
▼

修复循环 （循环回到 3/4）
命名缺陷 → 定向修复 → 单层重验。规则：一轮一个缺陷、diff 库存、不盲重建。上限 3 轮，超限问用户。
提示词：SKILL.md Debug Discipline（5 条反 flail 规则）
⚠ 软约束：3 轮上限
▼

提交 houdini_commit_sandbox
运行双硬门（模块化结构 + 朝向）作为最终检查。任一失败拒绝提交。bypass 需 skip_orientation/skip_structure + 文档化理由。
🔒 双硬门禁

失败处理 houdini_discard_sandbox / houdini_collect_diagnostics
诊断先于删除；策略性错误（拓扑破裂、多组件朝向错）才 discard + 全量重建。

4风险审计

基于最新代码状态的 8 项风险与优化点。已修复项标注 ✓，待办项按优先级分级。

R1 · 提示词冲突：harness.ts 残留 probe 指引 ✓ 已修复
harness.ts:69 曾教 agent "PROBE parameter names first (create + inspect + destroy)"， 与 C 站 + SKILL.md 的新流程矛盾。agent 在 sandbox 流程会看到这条，可能走回手动 probe。
修复：commit 18782ab → 改为指向 houdini_node_parms
R2 · Recipe 阶段无强制约束 🔴 高
Recipe 是纯思考阶段，无工具强制 agent 先提交。若 agent 跳过 recipe 直接 run_python_sandbox，component_id / orientation 门禁形同虚设—— 门禁只在 commit_sandbox 时跑，而用户可能直接接受未 commit 的 sandbox。
建议：考虑 recipe 校验前置工具，或 sandbox 也跑门禁预览
R3 · 视觉模型不可用时无降级路径 🔴 高
Layer 3 强依赖 describe_image。SKILL.md 写 "Do NOT skip describe_image"， 但视觉模型配置错误 / API 失败时（历史多次出现，见第24阶段），agent 会卡住。 没有明确的"视觉不可用时，确定性层全过能否 commit"的指引。
建议：增加视觉降级策略文档 + 确定性层全通过时的 commit 放行规则
R4 · Builder vs network_mode 边界模糊 🟡 中
Agent 须判断"能否表达为 recipe"，但 recipe schema 边界无清晰文档——只说 "non-standard topologies"。 带布尔切割、动态细分、条件分支的资产算不算？资深 TD 也会困惑，导致 agent 强行用 recipe 表达 不适合的拓扑，或在两者间反复横跳。
建议：补充 recipe 表达力边界文档 + 典型反例
R5 · PCA 回退路径的自洽错误风险 🟡 中
Agent 同时写组件代码 + expected_axis 时，错误是相关的——误解"X轴水平"会同时生成躺平轮子 + 写 expected_axis:X，PCA"愉快确认"。B 站用 construction_axis 解决了构造路径， 但 PCA 回退路径（老 recipe / 未声明 construction_axis）仍有此风险。
建议：强制 construction_axis，或对 PCA 路径加 crosscheck 告警
R6 · Manifest 重复加载无缓存 🟡 中
load_node_parms_manifest 每次调用都重新读 6MB 文件 + json.parse。 单次可接受，但 _validate_recipe 对每个 postprocess 节点都调一次 manifest_parm_names（每次都 load），多个节点会重复加载。
建议：加模块级缓存（load 一次，invalidate 靠版本号）
R7 · network_mode OUT 命名陷阱 🟢 低
network_mode 要求 OUT 节点名为 "OUT"，agent 可能命名 "out"/"OUTPUT"。 harness 有自动发现逻辑（找最大 component_id 节点），但这是隐式的，agent 不知道。
建议：Forbidden Patterns 补充 OUT 命名陷阱
R8 · SKILL.md 过长（1250+ 行） 🟢 低
整个 skill 注入 system prompt，对每次程序化任务都消耗大量 token。速查表（14 个 SOP） 现在可被 houdini_node_parms 替代，可考虑移到工具查询以大幅瘦身。
建议：速查表移到按需查询，SKILL 只保留触发指引

5双视角评估

同一套系统，普通用户与资深 TD 看到的是不同的图景。

👤普通用户视角

优点
  • "说一句话就出模型"——recipe → 自动建网 → 自动验证 → 提交，几乎不用懂 Houdini
  • 三层验证 + 门禁让结果可信任：不会提交一个轮子躺平的自行车
  • Debug Discipline 防止 agent 无脑重建，不浪费用户时间
  • C 站让参数名不再靠猜，减少了"做出来但法线不对"这类隐蔽缺陷
痛点
  • 失败时很被动：agent 跑 3 轮放弃后说"请你帮忙"，但用户可能不懂 Houdini。缺少"降级交付"机制
  • 等待时间长：build + 4 视图 + 视觉分析约 1-2 分钟，修复 ×3 约 5 分钟，无进度反馈
  • 视觉判断用户无法干预：describe_image 说 STRUCTURAL_DETAIL=2 要修，用户可能觉得够了，agent 却固执继续

🛠️资深 TD 视角

优点（架构专业）
  • 声明式 Builder 方向正确——把 createNode/wiring 易错操作换成确定性建网，解决"agent 不会写 Houdini API"根因
  • 门禁分层精准：确定性层权威廉价，视觉仅 advisory——避免用昂贵视觉模型做不擅长的朝向判断
  • construction_axis（B站）是亮点：PCA 估计变 ground truth，消除 AI 特有的自洽错误
  • sandbox 隔离 + commit 门禁符合 TD "先验证再落地"的工作习惯
  • C 站 manifest 把参数名漂移问题彻底解决——真实 H21 数据，端到端验证 cuspangle
担忧
  • recipe 表达力受限是硬伤：真实生产资产（布尔切割、变截面、条件细分、UV）很难塞进固定拓扑，TD 被迫回到 network_mode 手写
  • 缺少 HDA 封装路径：TD 核心工作流是"做完→封装 HDA 复用"，但 create_hda 没和 procedural pipeline 打通
  • manifest 是死快照：Houdini 升级/装新插件后需手动重跑生成脚本，无自动版本漂移检测
  • 没有黄金范例参考（D 站未做）：agent 每次从零推理，质量天花板受限
