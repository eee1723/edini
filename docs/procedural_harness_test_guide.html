这批具体做了什么

下面这些是你可以在行为上观察到的变化，不只是代码文件变多。

1. 沙盒执行

新增 edini/harness.py 和打包副本。程序化生成先创建唯一 job id 和 `/obj/edini_sandbox_*` 根节点，代码在这个沙盒上下文里执行。

2. 输出可序列化

脚本可以把节点、组件信息、统计结果写入 result。harness 会把 Houdini 对象转成路径或安全字符串，避免工具响应 JSON 崩掉。

3. 失败保留现场

执行失败时返回原始 error、traceback、stdout、stderr、diagnostics。默认保留沙盒节点，让你能继续看失败现场。

4. 结构验证

houdini_verify_asset 检查几何是否存在、最小 point/primitive 数、bounds 是否非零，以及 node errors 是否为空。

5. 生命周期工具

通过 houdini_commit_sandbox 改名提交到正式节点，或用 houdini_discard_sandbox 丢弃沙盒。提交会处理重名、非法名称和 display flag 错误。

6. Pi 工具暴露

在 Pi extension 里注册了 harness 工具 schema：sandbox、diagnostics、verify、commit、discard、safe viewport capture。

7. Skill 行为约束

procedural-modeling skill 现在明确要求先用 harness，不要直接 raw run，不要先删失败节点，不要探索 Qt/viewport 私有接口。

8. 安全截图路径

houdini_capture_viewport_safe 统一走受支持的 viewport capture。捕获失败时应该报告失败阶段，而不是乱试 UI 内部对象。

9. 梯子事故回归

加入 ladder regression 文档和测试，用“先沙盒、组件元数据、非空几何、非零 bounds、再提交”的目标防止旧问题复发。

期望流程

你和模型对话时，可以直接看它的工具调用是否沿着这条路走。

先说 recipe

说明要做什么资产、用 Python SOP/VEX/node network 哪个 backend、预期有哪些结构检查。

沙盒生成

调用 houdini_run_python_sandbox，现场出现唯一命名的 sandbox root。

诊断和验证

失败先诊断；成功也要验证 point/prims/bounds/node_errors。

安全截图

用 safe capture 看视觉结果。失败时报告 capture stage，不该探索 Qt widget。

提交或丢弃

验证通过才 commit 成正式节点；不满意就 discard，失败现场先留给你看。

手测重点

每项都可以标记状态，备注会自动保存在浏览器 localStorage。建议按顺序测。

失败信号

看到这些现象，就说明 harness 行为或 skill 约束可能没有生效。

模型一上来就用 houdini_run_python 直接改正式节点，没有说明沙盒、验证或提交步骤。
Python SOP/VEX 报错后马上删除失败节点，导致你看不到 traceback、node errors、参数或几何状态。
只说“看起来成功”，没有 point count、primitive count、bounds、node_errors 这些结构证据。
截图失败后开始探 Qt widgets、main window、viewport internals，而不是干净返回 capture failure。
commit 前没有调用 verify，或 verify 失败仍把沙盒改名成正式资产。
工具响应里出现 JSON serialization 报错，尤其是 result 里包含 Houdini node 对象时。
