🎯 核心原则

Code-first approach: 当用户需要程序化结果时，编写和执行代码（Python SOP 或 VEX），而不是手动创建大量节点。代码迭代更快，更容易参数化，Agent 可以自我验证。

⚠️ 关键发现

LLM 从零生成 VEX 代码的执行成功率仅 36%（HoudiniVexBench 基准测试）。因此必须使用模板适配策略而非从零生成。

⚙️ VEX 指导原则

  1. 短片段 — 一个 wrangle = 一个操作。5 个变换 → 5 个 wrangle 串联执行。
  2. 模板模式 — 基于已知构建块适配，而非从零编写：
    • 噪声位移：@P += normalize(@N) * noise(@P * chf("freq")) * chf("amp")
    • 密度散射：if(rand(@ptnum) > chramp("density", fit01(@P.y, 0, 1))) removepoint(0, @ptnum)
    • 方向对齐：p@orient = quaternion(dihedral({0,1,0}, normalize(@vel)))
  3. 验证 — 执行 VEX 后，始终调用 houdini_inspect_geo 检查点数、属性、包围盒。
  4. 失败两次 → 切换 Python SOP — Python 对复杂逻辑远比 VEX 可靠。

🐍 Python SOP 指导

优先创建持久化 Python SOP 节点而非临时的 houdini_run_python，这样结果保留在场景中且可参数化：

# 通过 houdini_run_python — 创建一个持久化 Python SOP
geo_node = hou.node("/obj").createNode("geo", "procedural_result")
py = geo_node.createNode("python", "generate")
py.parm("python").set("node = hou.pwd()\ngeo = node.geometry()\n# 生成代码在此")

Python SOP 适用场景：递归（L-system、树木）、外部数据、复杂循环、CSG、网格操作。

🔄 工作流

  1. 理解意图 — 什么形状/图案/效果？哪些参数需要可调？
  2. 选择方法 — VEX（逐元素）/ Python SOP（算法化）/ 节点网络（标准操作）
  3. 增量构建 — 一步一步来，每步后 houdini_inspect_geo 验证
  4. 视觉验证 — houdini_capture_viewport + describe_image 检查视觉效果
  5. 参数化 — VEX 用 ch()/chramp()/chv()，Python SOP 用 spare 参数

🐛 常见 VEX 陷阱

  • rand() 返回 float；需要向量时使用 set(rand(s), rand(s+1), rand(s+2))
  • 没有 float3 — 使用 vector 加 set()
  • foreach 语法：foreach (elem; array) { ... }
  • 矩阵乘法顺序：P * M 而非 M * P（列优先）

🎨 Copernicus（程序化贴图）

  1. 通过 houdini_run_python 在 /img 上下文中创建节点
  2. 优先使用 Copernicus 节点（copernicus::noise/ramp/math/merge）而非旧版 COP2
  3. 通过 sopimport COP 节点导入 SOP 数据
  4. 通过 hou.node(...).parm("execute").pressButton() 在 ROP 上烘焙贴图
