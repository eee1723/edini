技能 · project-modeling

项目化建模 — 通过锚点端口协作的组件

将多部件物体构建为一个 Project HDA:声明组件、搭骨架(各自带输出端口的空子网),然后在每个组件子网内自由建模。

程序化模型构建为 Project HDA:一个自包含的资产,其内部网络由一组组件子网构成。每个组件是一个子网,通过输出端口暴露自己的几何体和命名锚点。组件之间彼此协作:一个输出锚点,下一个消费锚点来确定自己该放在哪里。整个模型存在于一个 HDA 中,可手工编辑,且每个参数都保持实时联动。

这是构建任何多部件物体的默认方式。仅当模型是简单的"单一主体 + 叶子节点"且不需要组件拆分时,才使用 build_assembly(rooted-modeling)。

⛔ 护栏须知(动手前必读)

⚡ 头脑风暴应走快速通道

建模任务允许头脑风暴,但应作为快速通道:先问 1-2 个简短问题(风格?尺寸?),然后交给本技能的工作流。不要让头脑风暴跑完完整的"软件规格访谈"(5-10 个问题 + 设计文档 + 计划)—— 那个流程是为写代码准备的,不是为 3D 模型。头脑风暴技能本身现在带有一个"Houdini 程序化建模 — 快速通道豁免"小节;如果你读到它,请遵循该快速通道。

⛔ 不要把组件声明为独立的

当一个组件物理上依赖另一个组件(例如桌腿依赖桌面的锚点)时,你必须在搭骨架阶段的 ports.in 中声明该依赖。缺少它,构建器不会创建输入端口,下游组件就无法消费锚点 —— 你最后只能用 transform 节点手工摆放,这会毁掉参数化设计。详见下文"声明跨组件依赖"。

核心思想:组件 + 锚点端口

Project HDA 核心(edini::project SOP HDA) ├─ tabletop/ (子网,id="tabletop") │ ├─ <你的建模节点> │ ├─ out_geometry → output_0 [out[0] = 主几何体] │ └─ out_anchors → output_1 [out[1] = 锚点云] │ 点携带 @name,例如 "leg_mount_fr" ├─ legs/ (子网,id="legs") │ ├─ in_tabletop_leg_mount_fr (null,构建器自动创建 = 上游锚点) │ ├─ <你的建模:把桌腿放在锚点处> │ └─ out_geometry → output_0 └─ OUT (核心输出:所有组件合并)
  • out[0] 始终是组件的主几何体(它会合并到核心的 OUT)。
  • out[1+] 是锚点云:点携带 @P(位置)、@orient(四元数)和 @name(锚点身份 —— 例如 leg_mount_fr)。这就是一个组件告诉另一个组件"把东西放在这里"的方式。
  • 下游组件通过连接到上游子网的输出端口 1(output_index=1)来消费上游锚点。

工作流程(确定性步骤)

智能体通过三个专用工具加上标准节点工具来驱动此流程。这是唯一的建模路径 —— 不再有 build_assembly。

1. 创建项目 — project_create

project_create(name="project_table", goal="a small table")
  → 返回 { core_path: "/obj/project_table/project_core", ... }

首先调用此工具。工作区感知:如果用户在网络编辑器中选中了一个 Project HDA 核心,project_create 会复用它(返回其路径,不新建)—— 因此"选中一个 Project HDA → 在其中工作"是自然的工作流。仅在没有选中相关对象时才新建。请记住返回的 core_path。

2. 声明组件 + 搭骨架 — project_build_scaffold

传入 core_path 和 components(拆分方案)。每个组件包含一个 id(= 子网名)、purpose(用途)和 ports(端口)。

project_build_scaffold(core_path="/obj/project_table/project_core", components=[
  { "id": "tabletop", "purpose": "桌面,输出四条腿的安装锚点",
    "ports": { "out": [
        { "index": 0, "kind": "geometry", "description": "桌面几何" },
        { "index": 1, "kind": "anchors", "points": [
            { "name": "leg_mount_fr", "role": "mount" },
            { "name": "leg_mount_fl", "role": "mount" },
            { "name": "leg_mount_br", "role": "mount" },
            { "name": "leg_mount_bl", "role": "mount" } ] } ] } },
  { "id": "legs", "purpose": "四条桌腿,消费桌面锚点定位",
    "ports": { "out": [ { "index": 0, "kind": "geometry" } ],
                "in": [
                  { "from": "tabletop", "port": 1, "anchor": "leg_mount_fr" },
                  { "from": "tabletop", "port": 1, "anchor": "leg_mount_fl" },
                  { "from": "tabletop", "port": 1, "anchor": "leg_mount_br" },
                  { "from": "tabletop", "port": 1, "anchor": "leg_mount_bl" } ] } }
])

构建器会确定性地、幂等地创建:

  • 每个组件一个子网(tabletop/、legs/)—— id = 子网名
  • 每个子网内部:out_geometry + out_anchors(null 节点)→ output_0/output_1
  • 对每个 ports.in[]:外部连线 + 内部 in_<from>_<anchor> null 节点
  • 一个核心 OUT(合并所有组件几何体)并带显示标志
  • 核心参数面板上有一个"💬 与 Edini 对话"按钮(点击 → 弹出聊天框)

构建器不构建几何体或参数 —— 那是你的工作,自底向上完成。

⚠️ 声明跨组件依赖(关键 —— 不可跳过)

如果组件 B 物理上依赖组件 A(B 需要 A 的锚点来知道把自己放在哪),你必须在 B 的 ports.in 中声明。缺失 ports.in 是参数化模型损坏的头号原因 —— 没有它,构建器就不会创建 in_<from>_<anchor> 输入端口,导致 B 无法消费 A 的锚点,你被迫用 transform 节点手工摆放 B(这在 A 的参数变化时会失效)。

错误把桌腿声明为独立 —— 它们会漂浮在空间中,断开连接

{ "id": "legs", "ports": { "out": [{"index":0,"kind":"geometry"}], "in": [] } }

正确桌腿消费桌面的锚点 —— 构建器会连线 in_tabletop_<anchor>

{ "id": "legs", "ports": {
    "out": [{"index":0,"kind":"geometry"}],
    "in": [
      { "from": "tabletop", "port": 1, "anchor": "leg_mount_fr" },
      { "from": "tabletop", "port": 1, "anchor": "leg_mount_fl" },
      { "from": "tabletop", "port": 1, "anchor": "leg_mount_br" },
      { "from": "tabletop", "port": 1, "anchor": "leg_mount_bl" } ] } }

规则:在调用 project_build_scaffold 之前,先在纸上画出依赖图(桌面 → 桌腿,桌面 → 围板)。每条箭头都对应一个 ports.in 条目。如果某个组件没有上游依赖(很少见 —— 只有根节点如此),它的 in 可以是 []。

2b. 声明设计参数(推荐 —— 自顶向下,自动创建)

把 design_params 传给 project_build_scaffold,它们会自动作为备用参数(spare parms)创建在核心节点上。这是推荐路径 —— 你不需要 project_promote_params,也不需要手工创建备用参数。

project_build_scaffold(core_path="...", components=[...], design_params=[
    {"name":"length",   "label":"桌面长度",  "default":1.2,  "min":0.4, "max":3.0},
    {"name":"width",    "label":"桌面宽度",  "default":0.6,  "min":0.3, "max":1.5},
    {"name":"height",   "label":"桌高",      "default":0.75, "min":0.3, "max":1.2},
    {"name":"top_thick","label":"面板厚度",  "default":0.04, "min":0.01,"max":0.1},
    {"name":"leg_thick","label":"腿粗",      "default":0.05, "min":0.02,"max":0.2}
])
# → spare 参数 length/width/height/top_thick/leg_thick 自动创建在核心上。
# → 返回值包含 "design_params_created": 5

搭好骨架后,在组件子网内部的几何节点上引用这些参数,使用绝对路径(跨子网嵌套时相对 ch("../") 不可靠):

# 在 tabletop/body_box(tabletop 子网内的一个 box SOP)内部:
houdini_set_params_batch(node_path=".../tabletop/body_box", params={
    "sizex": "ch('/obj/.../project_core/length')",       # 绝对路径
    "sizey": "ch('/obj/.../project_core/top_thick')",
    "sizez": "ch('/obj/.../project_core/width')",
    "ty":    "ch('/obj/.../project_core/top_thick')*0.5"
})
⚠️ 使用绝对 ch() 路径

使用绝对 ch() 路径(例如 ch('/obj/geo1/project1/length')),不要用相对路径(ch('../../length'))。跨子网嵌套层级的相对路径经常产生"Bad parameter reference"警告并得到零几何体。核心路径在每条消息的 [Current Houdini Context] 块中给出 —— 用它。

3. 在每个组件子网内建模 — 标准节点工具

使用 houdini_create_node、houdini_connect_nodes、houdini_set_param。在每个子网内自由建模。参数是建模的副产品:当你需要一个可调值时,在子网上添加备用参数(通过对子网节点调用 houdini_set_param,或用 addSpareParmTuple),然后用 ch("../<parm>") 在几何节点中引用它。

构建主几何体(例如桌面 box,带可调长度)

# 1. 在子网上添加备用参数(参数是建模的副产品)。
houdini_set_param(node_path="<tabletop 子网>", "length", 2.0)  # 创建备用参数
# 2. 构建引用子网参数的几何体。
houdini_create_node(node_type="box", parent_path="<tabletop 子网>")
houdini_set_param(node_path, "sizex", 'ch("../length")')   # 实时引用子网参数
houdini_set_param(node_path, "sizey", 0.05)
houdini_connect_nodes(from_path=box_path, to_path="<tabletop>/out_geometry")

程序化地发射锚点(不要硬编码!)— project_add_anchors

# 锚点必须从几何体中派生(这样参数变化时它们会跟着移动),
# 绝不能硬编码 addpoint 坐标。project_add_anchors 生成一个实时
# VEX wrangle,在每次 cook 时测量组件的包围盒。
project_add_anchors(core_path="<core>", component_id="tabletop", anchors=[
    { "measure": "bbox_corner", "axes": "+X-Y+Z", "name": "leg_mount_fr" },
    { "measure": "bbox_corner", "axes": "-X-Y+Z", "name": "leg_mount_fl" },
    { "measure": "bbox_corner", "axes": "+X-Y-Z", "name": "leg_mount_br" },
    { "measure": "bbox_corner", "axes": "-X-Y-Z", "name": "leg_mount_bl" }
])
# 现在调整桌面尺寸(改核心 'length')→ 包围盒变化 → 锚点重新计算。

可用的测量方式:bbox_corner(需要 axes 如 "+X-Y+Z")、bbox_face_center(需要 face 如 "-Y")、bbox_center、grid_on_face(需要 face、rows、cols)、point_on_edge、array。每种都会发射带 @name 标签的点。

消费锚点(在下游组件中,例如桌腿)

# 构建器已经把 in_tabletop_leg_mount_fr 连到了上游锚点。
# 只要在它下游构建即可。
houdini_create_node(node_type="tube", parent_path="<legs 子网>")   # 一条桌腿
houdini_set_param(node_path, "rad", [0.04, 0.04])     # 向量参数
houdini_set_param(node_path, "height", 'ch("../leg_height")')  # 实时表达式
houdini_create_node(node_type="copytopoints", parent_path="<legs 子网>")
houdini_connect_nodes(from_path=tube_path, to_path=ctp_path, input_index=0)
houdini_connect_nodes(from_path="<legs>/in_tabletop_leg_mount_fr", to_path=ctp_path, input_index=1)
houdini_connect_nodes(from_path=ctp_path, to_path="<legs>/out_geometry")   # 主几何体输出

要直接获取上游组件的锚点端口(而非通过构建器的 in 节点),使用 output_index:

houdini_connect_nodes(from_path="<tabletop>", to_path="<something>", input_index=0, output_index=1)
# output_index=1 = tabletop 的锚点云 (out[1]);0 = 它的主几何体

4. 把参数提升到核心(自底向上)— project_promote_params

建模完成后(子网备用参数已存在并测试过),把它们提升到核心 HDA,这样整个模型就能从一个地方调节:

project_promote_params(core_path="<core>")
# 扫描每个子网的 spareParms() → 在核心上创建 <component>_<parm>
# (分组到 <component> 文件夹下,复制 min/max)→ 把子网参数重连为
# ch("../<component>_<parm>"),这样由核心驱动它。

自底向上:参数源自子网(你在那里建模 + 测试),然后被提升到核心以实现统一控制。核心成为调节入口;子网通过 ch() 引用跟随。

参数实时引用(自底向上,提升后由核心驱动)

参数起源于组件子网(建模时添加的备用参数)。执行 project_promote_params 后,每个参数被提升到核心,命名为 <component>_<parm>(分组、带 min/max),子网参数被重连为 ch("../<component>_<parm>") —— 于是核心驱动该值。

你所在的位置引用什么示例
子网内部的几何节点子网的参数tabletop/ 中的 box 用 ch("../length")
用户(提升后)核心的 <component>_<parm>用户设置 tabletop_length → 子网跟随 → 几何体 + 锚点更新

houdini_set_param 接受包含 ch(...) 的表达式字符串,并自动路由到 setExpression —— 因此实时参数能通过该工具正常工作。

何时使用

这是多部件程序化模型的唯一路径

build_assembly 已不再作为工具存在 —— 不要尝试调用它。任何由可拼接部件组成的物体(桌子、汽车、自行车、键盘、机器、建筑)都作为 Project HDA 在此构建。

情形使用
任何多部件物体(桌子=桌面+桌腿,汽车=车身+轮子,键盘=托盘+按键) Project HDA(本技能):project_create → project_build_scaffold → 在子网内建模 → project_promote_params
用户可能手工编辑的长期可编辑模型 Project HDA
单一生成器 / 一次性 SOP(无组件) houdini_run_python_sandbox

即使是"简单"的桌子也是多部件物体(桌面 + 桌腿):把它拆解成组件并在此构建。组件拆分正是让模型可理解、可编辑(以及日后可检测漂移)的关键。

建模纪律(方向 —— 已用真实案例精炼)

  • VEX 优先:大多数需求用一个 wrangle + 原生 SOP 即可。不要动辄求助于 Python SOP。
  • 不要把所有东西堆进单个 Python SOP:如果你用 Python,把它分散到多个 SOP 中,不要全堆进一个。
  • VEX 难搞的地方用原生节点(sweep、fuse、boolean):用原生节点。
  • 测量,而非硬编码:位置来自上游锚点或测量值,绝不要手打坐标。这正是保持模型参数化的关键。

常见错误

错误修复
调整组件尺寸时锚点不动 你硬编码了 addpoint(x,y,z)。改用 project_add_anchors,让锚点从几何体测量并实时重算。绝不硬编码锚点坐标。
Box 读不到参数 在子网内部时,把参数加在子网上(对子网节点调用 houdini_set_param),然后用 ch("../length") 从几何体引用。
connect_nodes 连不到锚点云 传入 output_index=1(默认 0 = 主几何体)
核心上的参数没有 min/max 在子网上加备用参数时带上 min/max(FloatParmTemplate),提升时会把它们复制上去。
project_create 新建了 HDA 而非用选中的那个 project_create 会复用选中的 Project HDA 核心。要强制新建,先取消选中。
建好了几何体但核心 OUT 什么都没显示 你的几何体必须流入 out_geometry → output_0 → 核心 OUT 的 merge。

本技能支持的内容(以及路线图)

  • ✅组件子网,带 2 个输出端口(几何体 + 锚点),幂等骨架

  • ✅两层实时参数(子网 ../ + 核心 ./ 经由提升)
  • ✅工具:create_node / connect_nodes(带 output_index)/ set_param(向量 + 表达式)
