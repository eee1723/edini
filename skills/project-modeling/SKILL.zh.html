技能 · project-modeling

项目化建模 — 通过测量锚点协作的组件

将多部件物体构建为程序化模型(桌子、汽车、自行车、键盘、机器、建筑):声明组件,把它们 scaffold 成协作的子网,在每个子网内自由建模。组件之间通过测量锚点连接(绝不硬编码)。这是多部件建模的默认且唯一路径。

为何存在此技能

一个被丢进 Houdini 的 LLM,默认会用以下四种方式构建模型 —— 这四种方式都会破坏参数化:

  1. 硬编码坐标 —— 直接敲 addpoint(1.2, 0, 0.6),而不是从几何体中测量位置。模型第一次看起来是对的,但任何参数一变就崩了。
  2. 跳过声明的依赖 —— 用 transform 节点摆放组件,而不是把一个组件的锚点连进下一个,于是上游几何体一变,部件就漂浮断开。
  3. 把所有东西堆进一个 Python SOP —— 写一个没人读得懂、没法调试、没法手工编辑的 200 行脚本。
  4. 过早宣告"完成" —— 四条腿只建了两条就停了,因为没人告诉它模型还没完成。

本技能强制走那条能同时击败上述四种问题的确定性路径:scaffold(搭骨架)空的组件子网、从几何体中 measure(测量)锚点、通过锚点端口 wire(连线)组件,并在进入下一步前 verify(验证)每一步都满足完成标准。平台层(guards.py)会硬拒绝硬编码的锚点,因此下文的提示层规则与平台层拒绝通过同一套词汇相互加固。

⛔ 护栏须知(动手前必读)

1. 在 ports.in 中声明每一个跨组件依赖

如果组件 B 物理上依赖组件 A 的锚点,那么 B 的 ports.in 必须列出它消费的每一个锚点。缺少 ports.in 是参数化模型损坏的头号原因 —— 构建器不会创建输入端口,B 无法消费 A 的锚点,你被迫用 transform 节点手工摆放 B。这是本规则的唯一权威表述;下文工作流会回指它,而非重述。

2. 永远 measure(测量)—— 绝不硬编码坐标

锚点必须从几何体中测量(project_add_anchors),这样参数变化时它们才会跟着移动。绝不手打字面坐标,如 addpoint(0, {0.5,0,0}) 或 addpoint(0, set(0.225,0,0.225))。平台护栏会拒绝在 Project HDA 组件内使用字面坐标的 addpoint —— 你会看到一个 Refused: hardcoded-coordinate addpoint ... 错误。注意:用计算出的位置(set(i-base,...)*step、@P、chf(...))是允许的,那是生成程序化几何(网格、贴纸、scatter)的正常写法;只有手打数字坐标才会被拒。

3. 头脑风暴是快速通道,不是完整访谈

对于建模任务,问 1-2 个快速问题(风格?尺寸?),给出一个简短的组件拆分,然后交给本技能的工作流。不要跑头脑风暴的完整软件规格流程 —— 那个流程是为代码准备的,不是为 3D 模型。

核心词汇

这三个词贯穿本技能、工具描述和平台错误消息。把它们当作触发器来记,而不是当作散文:

  • scaffold —— 构建器所做的事:创建带输出端口和跨组件连线的空组件子网。确定性、幂等。构建器从不构建几何体。(步骤 2)
  • anchor —— 一个组件发射的命名点,让另一个组件知道该把自己放在哪(@name、@P、@orient)。组件间的总线。(步骤 2–3)
  • measure —— 锚点的产生方式:每次 cook 时从实时几何体派生,绝不手打坐标。(步骤 3,由平台强制)

工作流程(确定性步骤)

由三个专用工具加上标准节点工具驱动。每一步都以一个 ✅ 完成标准收尾 —— 未满足前不要进入下一步。

1. 创建项目 — project_create

project_create(name="project_table", goal="a small table")
  → returns { core_path: "/obj/project_table/project_core", ... }

首先调用此工具。它是工作区感知的:如果网络编辑器中选中了一个 Project HDA 核心,它会复用(返回其路径);仅在没有选中相关对象时才新建。请记住返回的 core_path。

✅ 完成标准

返回了 core_path 并已为下一步保存。如果它意外新建了 HDA,说明你有一个过期的选中项 —— 取消选中后重试。

2. 声明组件 + scaffold — project_build_scaffold

传入 core_path、components 和 design_params。每个组件包含一个 id(= 子网名)、purpose(用途)和 ports(声明它发射什么、消费什么)。

project_build_scaffold(core_path="/obj/project_table/project_core", components=[
  { "id": "tabletop", "purpose": "桌面，输出四条腿的安装锚点",
    "ports": { "out": [
        { "index": 0, "kind": "geometry", "description": "桌面几何" },
        { "index": 1, "kind": "anchors", "points": [
            { "name": "leg_mount_fr", "role": "mount" },
            { "name": "leg_mount_fl", "role": "mount" },
            { "name": "leg_mount_br", "role": "mount" },
            { "name": "leg_mount_bl", "role": "mount" } ] } ] } },
  { "id": "legs", "purpose": "四条桌腿，消费桌面锚点定位",
    "ports": { "out": [ { "index": 0, "kind": "geometry" } ],
                "in": [
                  { "from": "tabletop", "port": 1, "anchor": "leg_mount_fr" },
                  { "from": "tabletop", "port": 1, "anchor": "leg_mount_fl" },
                  { "from": "tabletop", "port": 1, "anchor": "leg_mount_br" },
                  { "from": "tabletop", "port": 1, "anchor": "leg_mount_bl" } ] } }
])

构建器会确定性地、幂等地创建:每个组件一个子网;每个子网内部 out_geometry + out_anchors(null 节点)→ output_0/output_1;对每个 ports.in[],一条外部连线 + 内部 in_<from>_<anchor> null 节点;一个核心 OUT(合并所有几何体);核心面板上一个"💬 对话"按钮。

调用之前,先画出依赖图(桌面 → 桌腿,桌面 → 围板)。每条箭头都是一个 ports.in 条目(见护栏 1)。

一个组件可以声明其朝向 axis(取 X/Y/Z/-X/-Y/-Z 之一;默认 Y)。verify_orientation 把它当作基准真值 —— 请声明与几何体实际生成方式相符的轴,而不是你希望的样子。

✅ 完成标准

每个声明的组件都有子网,每个 ports.in 条目都有一条连好的 in_<from>_<anchor> 输入 null,且工具未返回验证错误。重跑是安全的(幂等)。

2b. 声明 design_params(自顶向下,自动创建)

在同一个 project_build_scaffold 调用中传入 design_params —— 它们会自动作为备用参数(spare parms)创建在核心节点上。这是必需的,不是可选的:没有设计参数,几何体就没有可引用的东西,后续也没有什么可提升的。

project_build_scaffold(core_path="...", components=[...], design_params=[
    {"name":"length",   "label":"桌面长度",  "default":1.2,  "min":0.4, "max":3.0},
    {"name":"width",    "label":"桌面宽度",  "default":0.6,  "min":0.3, "max":1.5},
    {"name":"height",   "label":"桌高",      "default":0.75, "min":0.3, "max":1.2},
    {"name":"top_thick","label":"面板厚度",  "default":0.04, "min":0.01,"max":0.1},
    {"name":"leg_thick","label":"腿粗",      "default":0.05, "min":0.02,"max":0.2}
])
# → spare 参数 length/width/height/top_thick/leg_thick 自动创建在核心上。

在子网内部的几何节点中引用这些参数时,使用绝对 ch() 路径(ch('/obj/.../project_core/length')),而不是相对 ch("../") —— 跨子网嵌套的相对路径经常产生零几何体。

✅ 完成标准

每个设计参数都作为备用参数存在于核心上,且返回值包含与你数量相符的 design_params_created。

3. 在每个子网内建模 + measure 锚点

使用 houdini_create_node、houdini_connect_nodes、houdini_set_param。参数是建模的副产品:当你需要一个可调值时,在子网上加一个备用参数,然后在几何体中用 ch("../...") 引用它。

构建主几何体(例如引用设计参数的桌面 box)

houdini_create_node(node_type="box", parent_path="<tabletop 子网>")
houdini_set_param(node_path, "sizex", "ch('/obj/.../project_core/length')")  # 绝对路径
houdini_connect_nodes(from_path=box_path, to_path="<tabletop>/out_geometry")

程序化地 measure(测量)锚点(不要硬编码!)— project_add_anchors

project_add_anchors(core_path="<core>", component_id="tabletop", anchors=[
    { "measure": "bbox_corner", "axes": "+X-Y+Z", "name": "leg_mount_fr" },
    { "measure": "bbox_corner", "axes": "-X-Y+Z", "name": "leg_mount_fl" },
    { "measure": "bbox_corner", "axes": "+X-Y-Z", "name": "leg_mount_br" },
    { "measure": "bbox_corner", "axes": "-X-Y-Z", "name": "leg_mount_bl" }
])
# 调整桌面尺寸 → 包围盒变化 → 锚点重新计算。实时联动。

可用测量方式:bbox_corner(需要 axes)、bbox_face_center(需要 face)、bbox_center、grid_on_face(需要 face、rows、cols)、point_on_edge、array。

消费锚点(在下游组件中)— 构建器已经把 in_tabletop_leg_mount_fr 连好了;在它下游构建即可

houdini_create_node(node_type="tube", parent_path="<legs 子网>")
houdini_connect_nodes(from_path=tube, to_path=ctp, input_index=0)
houdini_connect_nodes(from_path="<legs>/in_tabletop_leg_mount_fr", to_path=ctp, input_index=1)
houdini_connect_nodes(from_path=ctp, to_path="<legs>/out_geometry")
✅ 完成标准

(a) 每个组件的 out_geometry 都有几何体流入;(b) 步骤 2 中 ports.out[1] 声明的每个锚点都通过 project_add_anchors 发射了;(c) verify_orientation 通过(无轴不匹配)。如果其中任何一条未满足,模型就没完成 —— 继续做。

4. 把参数 promote(提升)到核心 — project_promote_params

建模完成后,把子网的备用参数提升到核心,这样整个模型就能从一个地方调节:

project_promote_params(core_path="<core>")
# → 每个子网的备用参数变成核心上的 <component>_<parm>(分组),
#   子网参数被重连为 ch("../<component>_<parm>") —— 由核心驱动。
✅ 完成标准(实时联动保证)

把核心上一个已提升的参数设为新值,重新 cook,确认几何体 + 锚点都更新了。如果改一个核心参数却没有移动几何体,说明两层 ch() 链断了 —— 不要宣告完成。

何时使用

这是多部件程序化模型的唯一路径

任何由可拼接部件组成的物体(桌子、汽车、自行车、键盘、机器、建筑)都作为 Project HDA 在此构建。已不再有 build_assembly 工具。

情形使用
任何多部件物体(桌子=桌面+桌腿,汽车=车身+轮子) Project HDA(本技能)
用户可能手工编辑的长期可编辑模型 Project HDA
单一生成器 / 一次性 SOP(无组件) houdini_run_python_sandbox
